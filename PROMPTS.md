# PROMPTS — 작업 요청 템플릿

Claude에게 작업을 시킬 때 복사해서 채우는 템플릿 모음.
빈칸(`<...>`)만 채우면 맥락이 충분히 전달되도록 설계했다.
공통 전제: **[RULES.md](RULES.md)와 [ARCHITECTURE.md](ARCHITECTURE.md)를 지킬 것.**

---

## 🟢 기능 추가

```
[기능 추가] <기능 이름>

## 목적
<이 기능이 왜 필요한가 / 어떤 사용자 흐름을 만드는가>

## 요구사항
- <구체적 동작 1>
- <구체적 동작 2>
- <엣지 케이스 / 제약>

## 배치
- 어느 패키지에? <engine / table / rule / equity / stats / web>
- engine을 건드리는가? (건드린다면 순수성 유지 방법 명시)

## 완료 기준
- [ ] 단위테스트 추가 (<검증할 불변식>)
- [ ] `./gradlew test` 통과
- [ ] 필요 시 *View DTO로 리댁션 처리
- [ ] 관련 문서(PROJECT/TODO/README) 갱신
```

## 🔴 버그 수정

```
[버그 수정] <한 줄 증상>

## 재현
- 상황: <어떤 테이블 상태 / 액션 순서에서>
- 기대: <무엇이 일어나야 하는가>
- 실제: <무엇이 일어나는가 (에러 메시지·스택트레이스 첨부)>

## 의심 지점
<추측되는 파일/메서드 — 모르면 "모름", 조사부터 요청>

## 요구
- 원인을 먼저 설명한 뒤 고칠 것 (증상만 덮지 말 것)
- **재현 테스트를 먼저 추가**해 실패를 확인하고, 고친 뒤 통과시킬 것
- 회귀 방지: 인접 케이스도 테스트에 포함
```

## 🟡 리팩토링

```
[리팩토링] <대상 범위>

## 동기
<중복 / 가독성 / 결합도 / 성능 중 무엇을 개선하나>

## 불변 조건 (절대 바뀌면 안 되는 것)
- 외부 동작·API·테스트 결과는 그대로 유지
- 아키텍처 철칙(engine 순수성, 리댁션, 단일 라이터) 유지

## 요구
- 동작 변경 없음을 **기존 테스트 전량 통과**로 증명
- 커밋을 의미 단위로 쪼갤 것 (구조 이동과 로직 변경 분리)
- 큰 변경이면 먼저 계획을 제시하고 승인받은 뒤 진행
```

## 🔵 API / 엔드포인트 생성

```
[API] <STOMP 목적지 또는 HTTP 경로>

## 종류
- [ ] STOMP  (`/app/...` 수신 / `/user/queue/...` 또는 `/topic/...` 송신)
- [ ] HTTP   (`GET|POST /api/...`)

## 계약
- 요청: <필드/타입 — record DTO로>
- 응답: <필드/타입 — *View DTO, 리댁션 규칙 명시>
- 인증/주체: 액션 주체는 principal? (기본: 예)

## 규칙
- 도메인 객체를 직접 노출하지 말고 dto/*View로 변환
- 홀카드 등 민감 정보 리댁션 확인
- 에러는 `/user/queue/errors`(STOMP) 또는 적절한 상태코드(HTTP)로
- 통합테스트로 end-to-end 검증
```

## ✍️ 사용 팁

- 템플릿은 **최소 채움**으로도 동작하게 설계됨 — 모르는 칸은 "모름/조사 요청"으로 남겨도 된다.
- 큰 작업은 먼저 `[리팩토링]`·`[기능 추가]`의 "계획 먼저" 옵션으로 방향을 잡고 착수.
- 반복되는 지시(커밋 자동화, 질문 피드백 등)는 이미 메모리에 있으니 매번 쓸 필요 없다.
