# RULES — 개발 규칙

이 프로젝트에서 코드를 더할 때 지켜야 할 규칙. 새 코드는 **주변 코드처럼** 읽혀야 한다.

---

## 1. 아키텍처 규칙 (최우선)

- **`engine` 패키지는 순수 도메인이다.** Spring / WebSocket / DB / Jackson 타입을 import 금지.
  어기면 이 프로젝트의 핵심 논거("단위테스트로 100% 검증")가 무너진다.
- **의존 방향은 항상 아래로.** `web → table/rule/equity/stats → engine`. 역방향 의존 금지.
- **동시성은 `Table`이 전담.** 엔진 안에 락·synchronized·스레드를 넣지 않는다.
- **홀카드는 리댁션 후에만 전송선에 실린다.** 도메인 객체를 그대로 브로드캐스트하지 말고
  반드시 `dto/*View`로 변환. 상대 홀카드가 쇼다운 전에 나가면 버그.
- **액션 주체는 `principal.getName()`만 신뢰.** 요청 body의 playerId를 신뢰하지 않는다.

## 2. 코드 스타일

- **언어**: Java 21. 최신 문법(`switch` 표현식, `record`, `List.of`, 패턴 매칭) 적극 사용.
- **불변 우선**: 컬렉션 반환은 `List.copyOf` / `Collections.unmodifiableMap`. 방어적 복사.
- **DTO는 `record`로.** (`JoinRequest`, `ActionRequest`, `*View`)
- **주석은 한국어**로, "왜"를 설명한다. 코드가 이미 말하는 "무엇"은 반복하지 않는다.
  (기존 코드의 주석 밀도·톤을 그대로 따를 것 — 예: `HandEngine`의 설계 노트 주석)
- **검증은 빠르게 실패**: 잘못된 인자는 `IllegalArgumentException`, 잘못된 상태는
  `IllegalStateException`로 즉시 던진다. 메시지는 한국어로 사유를 명확히.
- Lombok은 이미 의존성에 있음 — 보일러플레이트에만 절제해서 사용.

## 3. 테스트 규칙

- **모든 도메인 로직은 단위테스트 동반.** `engine`은 Spring 없이 순수 JUnit으로 검증.
- 규칙성 검증은 **속성 테스트**로: 무작위 자동플레이 N판 후 불변식(칩 총량 보존, 정상 종료).
- 실시간 계층은 **통합테스트**로: 실제 WebSocket 위에서 착석→핸드시작→리댁션 end-to-end.
- 커밋 전 `./gradlew test` 통과 필수. 깨진 채로 커밋하지 않는다.

| 패키지 | 테스트 초점 |
|--------|------------|
| `eval` | 족보 분류·휠·키커·7장 best5 |
| `game` | 헤즈업 쇼다운, 폴드 무혈입성, 사이드팟, 규칙 위반 거부, 300판 칩보존 |
| `table`| 로비·리댁션·차례 검증 |
| `web`  | WebSocket end-to-end, 홀카드 리댁션 |

## 4. 커밋 규칙

- **의미 단위로 커밋**. 한 커밋 = 한 논리 변경. 기능+포맷팅 섞지 않는다.
- 커밋 메시지는 **무엇을 왜** 했는지 한 줄 요약(기존 로그 톤 참고):
  `Add StatsService + ROI leaderboard (VPIP/PFR/net)`
- 의미 단위가 완성되면 커밋·푸시는 **묻지 말고 자동 진행**(표준 승인).
- `main`에서 직접 실험 금지 — 위험 작업은 브랜치.

## 5. 스코프 규칙 (포트폴리오 범위)

- **완성도 > 기능 수.** 새 기능보다 기존 코어의 견고함을 우선한다.
- 외부 유료 서비스(GTO 솔버 등) 연동은 기본 제외.
- 영속 DB는 아직 도입 안 함(인메모리). 도입 시 `engine` 순수성 유지가 조건.
- 배포 서버가 저사양(Lightsail 2GB)임을 기억 — 무거운 연산은 요청당 상한을 둔다.
